## 🎯 UX north star — "the user always knows what's going on"

The 2026-05-23 session surfaced one theme above everything else: **whenever the user couldn't see what the agent was doing, they assumed it was broken** (verbatim: "user has no clue what is going on atp, just 7 mins doing nothing"). The fixes shipped this session (F8/F9/F10/F11) close the worst gaps, but the UI is still uneven — some surfaces narrate well, others go silent. The next tier of work needs to apply the F10 pattern (always-visible, ticks every second, explains gaps as normal) across every panel.

**Principles to ship to in every UX change below:**

1. **No silent surfaces.** Every panel that can show an empty / loading state must say *why* (`"runpod pod boots when run_experiment fires"`, `"waiting for first sub-RLM response"`) — never just a blank rectangle.
2. **Show what's coming next.** When a primitive finishes, the UI should preview what the root is likely to do next (`"next: verify_against_rubric (rubric scoring)"`) so the user follows the pipeline without reading code.
3. **Errors say what failed AND what will happen now.** Not just `Exception` — `pydantic ValidationError on field 'datasets'; the root will retry with a corrected dict on the next REPL turn`.
4. **Every long-running thing has a tick.** Seconds counters everywhere, not just on the LiveActivityStrip. Pulse a dot if the underlying signal is alive.
5. **Tooltips explain the why.** Each chip / metric / status pill has a tooltip — never assume the user knows the rlms paper or the primitive list.
6. **Same vocabulary everywhere.** "Primitive" / "iteration" / "sub-RLM" / "runpod pod" — pick one term per concept and use it in tooltips, error messages, log lines, all panels.

These principles drive U1–U7 below.

### U1. RunpodStatusChip in the header (was T4 — promoted)
**Why:** the runpod sandbox is silent for the entire first half of the run because pods are created lazily at `run_experiment`. The user has no way to know that's normal — looks broken.
**What:** `frontend/src/components/lab/rlm/runpod-status-chip.tsx` next to the iteration chip. States: `runpod: not yet — created at run_experiment`, `runpod: provisioning (Xs)`, `runpod: ready <pod_id>`, `runpod: executing (Xs)`, `runpod: stopping`, `runpod: destroyed`. Backend: new `/runs/<id>/runpod-status` route reading from `runpod_backend.list_pods(project_id)`.
**Effort:** ~1h.

### U2. Error chip that says what + why + what's next
**Why:** today's error surface is `result_summary=Exception` plus a red border on the primitive node. Users have no idea whether the run is recoverable or doomed.
**What:** when a primitive errors, the LiveActivityStrip and the NodeDetailSidebar both render a structured block:
```
ERROR · plan_reproduction
  Reason: pydantic ValidationError — field 'datasets[0]' expected dict, got str "MNIST"
  Recovery: root will retry on the next REPL turn (no action needed)
```
Depends on T2 (pydantic detail in SSE). Without T2 we can only show the exception type — half useless.
**Effort:** ~45 min after T2.

### U3. "What's next" hint in the LiveActivityStrip
**Why:** users want to anticipate, not just react.
**What:** the strip already says what's running now — add a thin second line: `next likely: <inferred>` based on the primitive sequence (after `plan_reproduction` → `implement_baseline`; after `run_experiment` → `verify_against_rubric`; etc.). Infer from a static dependency map in the hook. When uncertain, omit.
**Effort:** ~30 min.

### U4. Sticky pipeline phase indicator
**Why:** users lose track of "are we in `understand the paper` or `run experiments`?". The iteration counter helps but doesn't map to the conceptual phase.
**What:** a band below the LiveActivityStrip with 6 phases as connected steps (Ingest → Understand → Plan → Implement → Run → Verify), the current phase pulsing. Derive from the highest-numbered ok primitive seen so far.
**Effort:** ~1h.

### U5. Empty-state copy on every panel
**Why:** the REPL state rail says "no variables yet"; great. The exploration canvas shows just a "Paper" node before iter 1 with no explanation; bad. The rubric strip says "rubric score" with empty bars; vague.
**What:** audit every panel. Each empty state gets a one-line "why" plus an ETA: "Rubric scores appear after the first `verify_against_rubric` call (typically after implement_baseline + run_experiment)". Hand-author copy; no LLM generation.
**Effort:** ~1.5h (find + audit every panel).

### U6. Tooltip-everything pass
**Why:** chips and labels are opaque unless you've read the rlms paper.
**What:** every chip + metric label + status pill gets a `title` (and `aria-label` for screen readers) that explains what the value means. "iteration 3" → "Iteration 3 — the root model has emitted 3 REPL turns so far. The rlms paper expects 1-10 iterations for a successful reproduction."
**Effort:** ~1h.

### U7. Toast for run-level state changes
**Why:** `repl_iteration` events are subtle. `run_complete` lands silently if the user isn't looking. Errors that aren't recoverable should be impossible to miss.
**What:** add a `<Toaster>` component (or reuse if one exists). Emit on: every `repl_iteration` (subtle, info), every primitive `error` (warn), `run_complete` (success/partial/failure tone matching verdict), `final_report.json` ready (success with link). 5s autodismiss except errors which stay until clicked.
**Effort:** ~1h (plus picking a toast lib if none installed).

### U8. Test the UX in `prefers-reduced-motion`
**Why:** the new pulse animations + the future toasts will be unpleasant for users with motion sensitivity.
**What:** wrap every animation in `@media (prefers-reduced-motion: reduce) { animation: none; }`. Verify by toggling the macOS Reduce Motion preference and reloading.
**Effort:** ~20 min.

## 🚧 Next ship — high value, small surface

### T1. Expose all 6 registered root models in `/models` (the real F3)

**Today:** `backend/app.py:619-624` hardcodes `[{sonnet}, {opus}]`. The UI is stuck on those two even though the resolver supports `gpt-5`, `qwen3-coder`, `kimi-k2.5`, `claude`, `claude-oauth`, `qwen3-coder-featherless`, `azure-gpt-4o`.

**Fix:**
1. `backend/app.py` — replace the hardcoded list with `[{key, label, provider, credentialPresent} for key in ROOT_MODELS]`. Reuse the env-var probe pattern already used elsewhere (`_has_provider_credentials`) so the UI can grey-out models without credentials.
2. `frontend/src/lib/demo/demo-run-types.ts` — widen `DemoModelChoice` from `"sonnet" | "opus"` to `string`.
3. `frontend/src/components/lab/upload-view.tsx` — the model select already maps the backend list; nothing changes after the type widens.
4. Test: existing `use-rlm-run` tests still pass; add one assertion on the model list shape if there's an integration test that hits `/models`.

**Effort:** ~30 min. **Risk:** widening the type from union → string loses some type safety; mitigated by the backend being the source of truth and the resolver already validating.

### T2. Surface pydantic ValidationError detail in primitive_call SSE

**Today:** `binding.py:165` emits `result_summary=type(exc).__name__` only (`"ValidationError"`) so the UI shows useless "Exception" labels (twice now in this session — `detect_environment` early in prj_5b5fe266b0b83f3d, `implement_baseline` + `plan_reproduction` in prj_20457ea6673b5a32). The exception IS re-raised to the REPL so the root sees the full pydantic trace and adapts, but the user has no clue what went wrong.

**Fix:**
1. `binding.py` — special-case `pydantic.ValidationError`: build `{errors: [{loc, msg, type} for e in exc.errors()]}`, JSON-serialize, truncate to ~500 chars, emit as `result_summary`. Field paths and pydantic types are safe (no LLM/paper values).
2. Front-end RlmHeader / NodeDetailSidebar — render the JSON as a small "what failed" block when `status=error` and `result_summary` starts with `ValidationError`.
3. Add 1 test in `binding.test.py` if it exists, or a primitive-call test in the hook tests.

**Effort:** ~45 min. **Risk:** low — additive emission, backwards compatible.

### T4. `code/Dockerfile` is staged for runpod but no pod ever spawned in this run

**Today:** the active run shows 0 RunPod pods in the API even though `sandbox=runpod`, `build_environment` ran twice, and `Dockerfile` is on disk. `run_experiment` hasn't fired yet — the pod is created only when `run_experiment` is called. That's fine, but the user has no visibility into "pod will be created when". Combined with the `0 active pods` query I run from monitoring, this looks like runpod is broken when it isn't.

**Fix:** add a `RunpodStatusChip` next to the iteration chip in `RlmHeader` showing one of: `runpod: not yet — created at run_experiment`, `runpod: starting (Xs)`, `runpod: ready <pod_id>`, `runpod: stopping`, `runpod: destroyed`. Source: poll `/runs/<id>/runpod-status` (new backend route reading from the runpod_backend handle).

**Effort:** ~1 hr — new route + UI chip. **Risk:** medium — runpod_backend may not expose the handle in a clean shape; check first before writing the chip.

## 🧭 Design-discussion (won't ship without consensus)

### D1. Change `force_sandbox` config default from `"docker"` to `""`

The current `"docker"` default exists by deliberate choice — the field comment says "RunPod is disabled — Docker is the supported execution sandbox". Changing the default to `""` would re-enable per-run sandbox flags by default, which is what most local dev wants, but contradicts the existing "guarantee docker" intent.

**Question for maintainer:** is the docker-only guarantee still load-bearing on any deployment we know of? If not, flip the default and remove the special-casing in `.env`. If yes, keep the default and ship a louder warning when a `sandbox=runpod` request gets silently downgraded.

### D2. `.env REPROLAB_DEFAULT_SANDBOX=docker` vs `start.sh` exporting `runpod`

Cosmetic disagreement (F4). Either:
- Remove the `.env` line so `start.sh`'s `:-runpod` default fires unambiguously, OR
- Change `start.sh` to default `docker` so it matches the `.env`

Tied to D1 — handle in the same PR.

## 👀 Watch / no action

### W1. `implement_baseline` runs 22 minutes then errors with opaque `Exception`

Saw it once on prj_20457ea6673b5a32. Root self-recovered and started iter 3 immediately. Could be a sub-agent token-budget hit, an aclose hang in the SDK, or the sub-agent legitimately failed to write a valid baseline. T2 (above) will give us the error detail next time. Don't fix until we have evidence on the failure mode.

### W2. `demo_status.json::updatedAt` lags far behind real progress

Known-issues §10 — architectural. Backend writes status snapshots far less often than the SSE event stream, so the timestamp goes stale. The LiveActivityStrip (F10) deprioritizes this by giving the user a real-time signal that doesn't depend on `updatedAt`. Skip unless someone complains again.

### W3. The `0 active pods` snapshot during build_environment

This is correct — RunPod pods are only created at `run_experiment` time. T4 (above) addresses the visibility gap. The underlying behavior is right.
