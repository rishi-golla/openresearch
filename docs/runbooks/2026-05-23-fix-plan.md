## 🎯 UX north star — "the user always knows what's going on"

The 2026-05-23 session surfaced one theme above everything else: **whenever the user couldn't see what the agent was doing, they assumed it was broken** (verbatim: "user has no clue what is going on atp, just 7 mins doing nothing"). The fixes shipped this session (F8/F9/F10/F11) close the worst gaps, but the UI is still uneven — some surfaces narrate well, others go silent. The next tier of work needs to apply the F10 pattern (always-visible, ticks every second, explains gaps as normal) across every panel.

**Principles to ship to in every UX change below:**

1. **No silent surfaces.** Every panel that can show an empty / loading state must say *why* (`"runpod pod boots when run_experiment fires"`, `"waiting for first sub-RLM response"`) — never just a blank rectangle.
2. **Show what's coming next.** When a primitive finishes, the UI should preview what the root is likely to do next (`"next: verify_against_rubric (rubric scoring)"`) so the user follows the pipeline without reading code.
3. **Errors say what failed AND what will happen now.** Not just `Exception` — `pydantic ValidationError on field 'datasets'; the root will retry with a corrected dict on the next REPL turn`.
4. **Every long-running thing has a tick.** Seconds counters everywhere, not just on the LiveActivityStrip. Pulse a dot if the underlying signal is alive.
5. **Tooltips explain the why.** Each chip / metric / status pill has a tooltip — never assume the user knows the rlms paper or the primitive list.
6. **Same vocabulary everywhere.** "Primitive" / "iteration" / "sub-RLM" / "runpod pod" — pick one term per concept and use it in tooltips, error messages, log lines, all panels.

These principles drove U1–U8 below.

### U1. RunpodStatusChip in the header (was T4 — promoted)
**Closed:** `RlmHeader` now renders `RunpodStatusChip`. It shows the important silent states (`runpod: not yet`, `runpod: ready at experiment`, `runpod: executing`, terminal experiment outcomes) and polls `/runs/<id>/runpod-status` when a run is active. The backend route derives state from dashboard events first and queries the RunPod API only when a pod could actually exist.

**Verified:** frontend header test covers the pre-pod state; backend API test covers the route's lazy-creation explanation.

### U2. Error chip that says what + why + what's next
**Closed:** primitive errors now render as structured recovery blocks in `LiveActivityStrip` and `NodeDetailSidebar`:
```
ERROR · plan_reproduction
  Reason: Pydantic ValidationError — safe field/type detail from T2
  Recovery: root will retry, fall back, or repair on the next REPL turn
```

**Verified:** NodeDetailSidebar regression test asserts the reason/recovery rendering.

### U3. "What's next" hint in the LiveActivityStrip
**Closed:** the strip renders a thin `next likely:` line for predictable primitive transitions.

### U4. Sticky pipeline phase indicator
**Closed:** `PipelinePhaseStrip` now shows Ingest → Understand → Plan → Implement → Run → Verify with current/error phase styling.

### U5. Empty-state copy on every panel
**Closed:** REPL state, exploration canvas, rubric strip, report rail, and primitive history now explain why they are empty and when data appears.

### U6. Tooltip-everything pass
**Closed:** status, project, iteration, cost, phase steps, RunPod state, report stats, canvas, and primitive history now carry explanatory titles/labels.

### U7. Toast for run-level state changes
**Closed:** `RunToasts` emits non-blocking updates for new iterations, primitive errors, terminal run states, and final report readiness. Error toasts are sticky until dismissed; info toasts auto-dismiss.

### U8. Test the UX in `prefers-reduced-motion`
**Closed:** new pulse/entry animations are gated behind `prefers-reduced-motion: no-preference`; the existing inline live-pulse now disables itself under reduced motion.

## 👀 Watch / no action

### W1. `implement_baseline` runs 22 minutes then errors with opaque `Exception`

Saw it once on prj_20457ea6673b5a32. Root self-recovered and started iter 3 immediately. Could be a sub-agent token-budget hit, an aclose hang in the SDK, or the sub-agent legitimately failed to write a valid baseline. T2 (above) will give us the error detail next time. Don't fix until we have evidence on the failure mode.

### W2. `demo_status.json::updatedAt` lags far behind real progress

Known-issues §10 — architectural. Backend writes status snapshots far less often than the SSE event stream, so the timestamp goes stale. The LiveActivityStrip (F10) deprioritizes this by giving the user a real-time signal that doesn't depend on `updatedAt`. Skip unless someone complains again.

### W3. The `0 active pods` snapshot during build_environment

This is correct — RunPod pods are only created at `run_experiment` time. T4 (above) addresses the visibility gap. The underlying behavior is right.
